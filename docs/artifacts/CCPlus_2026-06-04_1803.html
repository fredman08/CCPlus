What's New

Highlights from Claude Code CLI v2.1.152 → v2.1.162 (late May – early June 2026)[1].

  • Claude Opus 4.8 (v2.1.154) — released with a high effort default; Fast mode on Opus 4.8 runs at 2× the standard rate for ~2.5× speed[1].
  • Dynamic workflows (v2.1.154) — Claude can orchestrate work across tens to hundreds of agents for substantive tasks[1].
  • ultracode setting (v2.1.160) — a session-only Claude Code setting (not a model effort level) that sends xhigh to the model and additionally orchestrates dynamic workflows for each substantive task; set via /effort or --settings[1][4].
  • Lean system prompt is now the default (v2.1.154), except on Haiku/Sonnet/4.7-era models[1].
  • Auto mode no longer requires opt-in consent (v2.1.152), and auto mode is now available on Bedrock, Vertex, and Foundry for Opus 4.7+ via CLAUDE_CODE_ENABLE_AUTO_MODE=1 (v2.1.158)[1].
  • Skills & plugins auto-load — plugins placed in .claude/skills load automatically, and claude plugin init <name> scaffolds a new plugin (v2.1.157)[1].
  • Review tooling — /code-review --fix applies findings to the working tree (v2.1.152), and /simplify became a cleanup-only review with auto-fix (v2.1.154)[1].
  • New safety prompts — Claude Code now prompts before writing to shell startup files and git config, and acceptEdits mode prompts before writing build-tool config files (v2.1.160)[1].
  • New hooks & skill reloading — a MessageDisplay hook can transform assistant messages, /reload-skills re-scans skill directories, and SessionStart hooks can set reloadSkills: true and a sessionTitle (v2.1.152)[1].
  • Agent view improvements — claude agents now shows done/total for fanned-out work and a PR column; Remote Control appears as a persistent footer pill (v2.1.161–162)[1].

Tips & Techniques

  • Bootstrap project memory with /init. It analyzes your codebase and writes a starter CLAUDE.md with build/test commands and conventions; if one exists it suggests improvements rather than overwriting. Set CLAUDE_CODE_NEW_INIT=1 for an interactive multi-phase flow that also offers to set up skills and hooks[3].
  • Use opusplan for plan-heavy work. It runs Opus during plan mode for reasoning/architecture, then automatically switches to Sonnet for execution[4].
  • Tune reasoning with /effort. Opus 4.8 supports low, medium, high (default), xhigh, and max. Drop one-off ultrathink into a prompt for deeper reasoning on that turn only[4].
  • Keep CLAUDE.md under ~200 lines and specific ("Use 2-space indentation", "Run npm test before committing"). Longer files consume more context and reduce adherence[3].
  • Scope instructions with .claude/rules/. Path-scoped rules (YAML paths: frontmatter) load only when Claude touches matching files, saving context vs. putting everything in CLAUDE.md[3].
  • Add the settings JSON schema for editor autocomplete: "$schema": "https://json.schemastore.org/claude-code-settings.json"[2].
  • Audit what Claude remembers with /memory — it lists loaded CLAUDE.md/rules files and lets you browse auto-memory (on by default since v2.1.59)[3].
  • Enforce hard rules with hooks, not prose. CLAUDE.md is context (not guaranteed compliance); for "must happen" actions like running a linter before commit, use a PreToolUse hook[3].
  • 1M-token context is auto-enabled for Opus on Max/Team/Enterprise plans; Sonnet 1M requires usage credits. Append [1m] to a model alias for long sessions (e.g. /model opus[1m])[4].

Hacks

Lesser-known but documented power-user techniques.

  • Leave maintainer notes for free. Block-level HTML comments (<!-- ... -->) in CLAUDE.md are stripped before the content enters context, so they cost zero tokens[3].
  • Import files into CLAUDE.md with @path syntax (e.g. @README, @package.json) instead of duplicating content; up to four levels deep[3].
  • Force a cheaper model for all subagents with CLAUDE_CODE_SUBAGENT_MODEL — it overrides per-invocation model choices for fan-out work (set to inherit to revert)[4].
  • Run background shell sessions with claude --bg --exec '<command>' (v2.1.154)[1].
  • Reuse an existing AGENTS.md. Claude reads CLAUDE.md, not AGENTS.md — create a CLAUDE.md that does @AGENTS.md so both tools share one source of truth (on Windows, prefer the import over a symlink)[3].

Recommended Default-Config Changes

A starter ~/.claude/settings.json (user scope) for a strong default setup. On a managed/Enterprise machine, managed settings take precedence and may restrict models or permissions — these are user-level suggestions that apply where policy allows[2][4].

{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "model": "opusplan",
  "effortLevel": "high",
  "outputStyle": "default",
  "statusLine": { "type": "command", "command": "~/.claude/statusline.sh" },
  "permissions": {
    "allow": [
      "Bash(git status)", "Bash(git diff *)", "Bash(git log *)",
      "Bash(npm run lint)", "Bash(npm run test *)"
    ],
    "deny": [
      "Read(./.env)", "Read(./secrets/**)", "Bash(curl *)"
    ]
  },
  "attribution": { "commit": "🤖 Generated with Claude Code" }
}

Notes: opusplan gives Opus-quality planning with Sonnet-speed execution[4]; on an Enterprise subscription seat the Default model is Sonnet 4.6, and Claude Code may fall back to Sonnet if you hit an Opus usage threshold[4]. outputStyle changes take effect after /clear or restart; model can also be switched mid-session with /model[2].

A starter CLAUDE.md

# Project: <name>

## Commands
- Build: <cmd>
- Test: <cmd>   # run before committing
- Lint: <cmd>

## Conventions
- <language/style rules — be specific, e.g. "2-space indentation">
- Source layout: <where handlers/components/tests live>

## Workflow
- Use plan mode for changes under <sensitive dir>.
- Commit frequently with clear messages.

Keep it under ~200 lines; move path-specific guidance into .claude/rules/ and repeatable procedures into skills[3].

Built-in slash commands worth knowing early

  • /init — generate or improve CLAUDE.md[3]
  • /model, /effort — switch model / reasoning depth[4]
  • /memory — view & edit loaded instructions and auto-memory[3]
  • /code-review (add --fix) and /simplify — review & cleanup[1]
  • /config, /status — settings & account/model info[2][4]

Good habits

  • Scope tasks tightly and use plan mode for risky changes.
  • Commit frequently; review Claude's diffs before accepting.
  • Add a correction to CLAUDE.md the second time you repeat it[3].
  • Prefer hooks for must-run steps; prefer skills for repeatable multi-step procedures[3].
